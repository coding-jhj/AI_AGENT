# 변경 내역 (v2 — 멘토 피드백 반영)

## 변경 전/후 비교

### P0 — 즉시 수정 (보안 + 한도 이슈)

#### 1. `_agent_cache` 제거 (`main.py`)

| 구분 | 변경 전 | 변경 후 |
|------|---------|---------|
| 코드 | `_agent_cache: dict = {}` 로 API 키 → Agent 매핑 캐싱 | 캐시 없음, 매 요청마다 `create_agent()` 호출 |
| 문제점 | 서버 메모리에 API 키가 잔류 | 요청 처리 후 즉시 소멸 (함수 스코프 종료) |
| 영향 | README와 실제 동작 불일치 (보안 버그) | 문서-코드 일치 |

**직접 설명 (제 언어로):**

캐시를 쓴 이유는 `create_agent()`가 반복 호출될 때마다 LangChain 초기화 비용이 든다고 생각했기 때문입니다.
그런데 실제로는 두 가지 문제가 있었습니다.

첫째, Groq LLM 호출이 수백ms~수초 걸리는 데 비해 Python 객체 생성 비용은 무시할 수준입니다.
성능상 이득이 거의 없는 최적화였습니다.

둘째 (더 중요), README에 "서버에 키를 저장하지 않습니다"라고 써놨는데,
`_agent_cache`의 key가 바로 API 키 문자열이었습니다.
즉 서버가 살아있는 동안 메모리에 키가 그대로 남아 있었습니다.
HuggingFace Spaces처럼 공유 환경에서 메모리 덤프가 일어나면 키가 노출될 수 있습니다.

캐시를 제거하면 API 키는 `query_agent()` 함수 스코프에서만 존재하고,
함수 종료 시 GC 대상이 됩니다. 이것이 README에 쓴 내용과 일치하는 동작입니다.

---

#### 2. `max_tokens` 축소 (`agent.py`)

| 구분 | 변경 전 | 변경 후 |
|------|---------|---------|
| 값 | `max_tokens=2048` | `max_tokens=1024` |
| 이유 | 기본값 그대로 사용 | Groq 무료 TPM 6,000 한도 대비 여유 확보 |

ReAct 루프는 `Thought → Action → Observation`을 반복할 때마다 이전 내용이 누적됩니다.
2048이면 루프 2~3회만에 분당 토큰 한도에 걸립니다.
1024로 낮추면 같은 TPM 내에서 루프를 2배 더 돌 수 있습니다.

#### 3. 모델 선택 드롭다운 (`main.py` + `index.html`)

- Llama 3.3 70B Versatile — 품질 우선
- **Llama 3.1 8B Instant — 속도/한도 우선 (기본값, 추천)**
